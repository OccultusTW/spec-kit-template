# 任務：[功能名稱]

**輸入**：來自 `/specs/[###-feature-name]/` 的設計文件
**先決條件**：plan.md（必需）、spec.md（使用者故事必需）、research.md、data-model.md、contracts/

**測試**：下面的範例包含測試任務。測試是選擇性的 - 僅在功能規格中明確要求時包含。

**組織**：任務按使用者故事分組，以實現每個故事的獨立實作和測試。

## 格式：`[ID] [P?] [Story] 描述`

- **[P]**：可以並行運行（不同檔案，無依賴性）
- **[Story]**：此任務屬於哪個使用者故事（例如：US1、US2、US3）
- 在描述中包含確切的檔案路徑

## 路徑慣例

- **單一專案**：`src/`、`tests/` 在儲存庫根目錄
- **網頁應用程式**：`backend/src/`、`frontend/src/`
- **行動**：`api/src/`、`ios/src/` 或 `android/src/`
- 下面顯示的路徑假設為單一專案 - 根據 plan.md 結構調整

## 依賴關係與執行順序

### 階段依賴關係

- **設定（階段 1）**：無依賴關係 - 可立即開始
- **基礎（階段 2）**：依賴於設定完成 - 阻塞所有使用者故事
- **使用者故事（階段 3+）**：全部依賴於基礎階段完成
  - 使用者故事可以並行進行（如有人力）
  - 或按優先級順序依次進行（P1 → P2 → P3）
- **打磨（最終階段）**：依賴於所有所需使用者故事完成

### 使用者故事依賴關係

- **使用者故事 1（P1）**：可在基礎（階段 2）後開始 - 對其他故事無依賴關係
- **使用者故事 2（P2）**：可在基礎（階段 2）後開始 - 可能與 US1 整合但應可獨立測試
- **使用者故事 3（P3）**：可在基礎（階段 2）後開始 - 可能與 US1/US2 整合但應可獨立測試

### 在每個使用者故事內

- 測試（如包含）必須在實作前撰寫並失敗
- 模型先於服務
- 服務先於端點
- 核心實作先於整合
- 故事完成後再移至下一個優先級

### 並行機會

- 所有標記為 [P] 的設定任務可以並行運行
- 所有標記為 [P] 的基礎任務可以並行運行（在階段 2 內）
- 一旦基礎階段完成，所有使用者故事可以並行開始（如果團隊容量允許）
- 故事中所有標記為 [P] 的測試可以並行運行
- 故事中標記為 [P] 的模型可以並行運行
- 不同的使用者故事可以由不同的團隊成員並行工作

## 實作策略

### MVP 優先（僅使用者故事 1）

1. 完成階段 1：設定
2. 完成階段 2：基礎（關鍵 - 阻塞所有故事）
3. 完成階段 3：使用者故事 1
4. **停止並驗證**：獨立測試使用者故事 1
5. 如果準備就緒則部署/展示

### 增量交付

1. 完成設定 + 基礎 → 基礎準備就緒
2. 添加使用者故事 1 → 獨立測試 → 部署/展示（MVP！）
3. 添加使用者故事 2 → 獨立測試 → 部署/展示
4. 添加使用者故事 3 → 獨立測試 → 部署/展示
5. 每個故事都能增加價值而不破壞先前的故事

### 並行團隊策略

有多個開發人員時：

1. 團隊一起完成設定 + 基礎
2. 一旦基礎完成：
   - 開發人員 A：使用者故事 1
   - 開發人員 B：使用者故事 2
   - 開發人員 C：使用者故事 3
3. 故事獨立完成和整合

## 注意事項

- [P] 任務 = 不同檔案，無依賴關係
- [Story] 標籤將任務映射到特定使用者故事以實現可追溯性
- 每個使用者故事應該可獨立完成和測試
- 在實作前驗證測試失敗
- 在每個任務或邏輯組後提交
- 在任何檢查點停止以獨立驗證故事
- 避免：模糊任務、相同檔案衝突、破壞獨立性的跨故事依賴關係
